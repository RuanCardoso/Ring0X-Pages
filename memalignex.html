01 — A Memória é uma fita de bytes

Cada endereço guarda 1 byte

Imagine a memória RAM como uma longa fita de caixinhas numeradas. Cada caixinha tem um endereço único e guarda exatamente 1 byte (8 bits). Variáveis maiores ocupam caixinhas consecutivas.

02 — O que é alinhamento?

O processador prefere endereços "alinhados"

O hardware lê a memória em blocos de tamanho fixo (4 ou 8 bytes de uma vez). Para ser eficiente, cada tipo de dado deve começar em um endereço múltiplo do seu tamanho.

char
1 byte — alinha em qualquer endereço
short
2 bytes — endereço múltiplo de 2
int / float
4 bytes — endereço múltiplo de 4
double / long
8 bytes — endereço múltiplo de 8
⚠️

Quando um dado não está alinhado, o processador precisa fazer duas leituras para montar o valor, desperdiçando ciclos de clock. Em alguns sistemas embarcados, gera até falha de hardware.

03 — Padding: o compilador insere bytes "vazios"

Bytes de preenchimento (padding) garantem o alinhamento

O compilador insere bytes extras entre os campos da struct para que cada campo comece em um endereço corretamente alinhado. Esses bytes não armazenam dados — são apenas "espaçadores".

struct Exemplo {
    char   a;   // 1 byte  — offset 0
    // ← 3 bytes de padding inseridos aqui!
    int    b;   // 4 bytes — offset 4 (múltiplo de 4 ✓)
    char   c;   // 1 byte  — offset 8
    // ← 7 bytes de padding no final!
    double d;   // 8 bytes — offset 16 (múltiplo de 8 ✓)
}; // sizeof = 24 bytes (não 14!)

04 — Ordem dos campos importa!

Reordenar campos pode reduzir o tamanho da struct

Colocar os campos maiores primeiro minimiza o padding necessário. Veja a diferença entre duas structs com os mesmos campos em ordens diferentes:

❌ Ordem ingênua

sizeof = 24 bytes
struct Ruim {
  char   a; // 1B
  // 3B pad
  int    b; // 4B
  char   c; // 1B
  // 7B pad
  double d; // 8B
};

✅ Ordem otimizada

sizeof = 16 bytes
struct Bom {
  double d; // 8B
  int    b; // 4B
  char   a; // 1B
  char   c; // 1B
  // 2B pad (final)
};
🏆

Regra de ouro: ordene os campos do maior para o menor tipo. Isso minimiza o padding e economiza memória — especialmente importante quando você tem arrays com milhões de structs.

05 — Tail padding: preenchimento no final

O tamanho da struct é arredondado para cima

Além do padding entre campos, o compilador adiciona bytes ao final da struct para que, em um array, o próximo elemento também comece alinhado. O tamanho final é sempre múltiplo do maior alinhamento da struct.

struct Tail {
    int  x;  // 4 bytes — offset 0
    char y;  // 1 byte  — offset 4
    // ← 3 bytes de tail padding!
};
// sizeof(Tail) = 8  (não 5!)
// Por quê? Para que Tail arr[2] funcione:
// arr[0] em 0x00, arr[1] em 0x08 (alinhado em 4) ✓
