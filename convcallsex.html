Como funções do Windows passam parâmetros, retornam valores e organizam a pilha — com exemplos reais da WinAPI como MessageBoxA e CreateFileW.

01 — Por que precisamos de uma convenção?

Funções precisam de um contrato em comum

Quando main() chama MessageBoxA() da user32.dll, as duas foram compiladas separadamente — talvez em anos diferentes, por compiladores diferentes. Para funcionarem juntas precisam de um contrato fixo: onde estão os argumentos? Onde vai o resultado? Quem limpa a pilha?

notepad.exe — chamando a WinAPI
#include <windows.h> int main() { // 4 argumentos — a convenção diz ONDE cada um vai MessageBoxA( NULL, // 1º arg: janela pai "Olá, Mundo!", // 2º arg: texto da mensagem "Meu Programa", // 3º arg: título da janela MB_OK // 4º arg: botões ); return 0; } // MessageBoxA está em user32.dll — já compilada pela Microsoft. // Ela PRECISA encontrar os args exatamente onde a convenção diz.
💡

A convenção de chamadas define: (1) onde ficam os argumentos (registradores ou pilha), (2) onde vai o valor de retorno, (3) quem limpa a pilha depois, e (4) quais registradores devem ser preservados. O Windows tem duas principais convenções que você precisa conhecer.

02 — Convenção 1: Microsoft x64 ABI (Windows 64-bit moderno)

Todo programa 64-bit no Windows usa esta convenção

Todo .exe e .dll de 64-bit no Windows — incluindo kernel32.dll, user32.dll e toda a WinAPI moderna — segue esta convenção. O compilador MSVC a aplica automaticamente.

Shadow Space (Home Space): Mesmo que os 4 primeiros args vão em registradores, o caller sempre reserva 32 bytes extras na pilha antes do CALL. São 4 slots de 8 bytes onde a função chamada pode "salvar" seus registradores de arg se precisar. Obrigatório mesmo que a função não use a área.

Exemplo: MessageBoxA em Assembly x64
Assembly gerado pelo MSVC — x64
; MessageBoxA(NULL, "Olá", "Título", MB_OK) main: sub rsp, 40 ; reserva shadow space (32B) + alinha pilha em 16B mov ecx, 0 ; 1º arg: hWnd = NULL → RCX lea rdx, msg ; 2º arg: lpText → RDX lea r8, titulo ; 3º arg: lpCaption → R8 mov r9d, 0 ; 4º arg: uType=MB_OK → R9 call MessageBoxA ; chama user32.dll ; retorno (IDOK = 1) estará em EAX / RAX add rsp, 40 ; CALLER limpa a pilha (desfaz o SUB) xor eax, eax ret

Se a função tem mais de 4 argumentos, o 5º em diante vai na pilha acima do shadow space, em ordem direta. Veja o exemplo com CreateFileW (7 args) na seção 05.

03 — Convenção 2: __stdcall (WinAPI 32-bit legada)

A convenção clássica de toda a WinAPI 32-bit

Antes do Windows 64-bit, toda a WinAPI usava __stdcall. Até hoje você a encontra em programas 32-bit: jogos antigos, aplicações legadas, muitos drivers e ferramentas de segurança. A grande diferença: tudo passa pela pilha, não por registradores.

__stdcall — regras

Win32 / x86 32-bit
  • 📌 Args vão na pilha, ordem inversa (último primeiro)
  • 📌 Callee limpa a pilha usando RET N
  • 📌 Retorno inteiro em EAX
  • 📌 EBX, ESI, EDI, EBP devem ser preservados
  • 📌 Nome decorado: _FuncName@NBytes

Microsoft x64 ABI — regras

Windows 64-bit moderno
  • 📌 4 primeiros args em RCX, RDX, R8, R9
  • 📌 Caller limpa a pilha com ADD RSP, N
  • 📌 Retorno inteiro em RAX
  • 📌 Shadow space de 32 bytes sempre obrigatório
  • 📌 Sem decoração de nome
Exemplo: MessageBoxA em Assembly __stdcall (x86 32-bit)
Assembly 32-bit — __stdcall (user32.dll)
; MessageBoxA(NULL, "Olá", "Título", MB_OK) — x86 32-bit ; Protótipo: int __stdcall MessageBoxA(HWND, LPCSTR, LPCSTR, UINT); main: ; Argumentos empurrados em ordem INVERSA (último → primeiro) push 0 ; 4º arg: MB_OK = 0 (empurra por último) push offset titulo ; 3º arg: lpCaption push offset msg ; 2º arg: lpText push 0 ; 1º arg: hWnd = NULL (empurra por primeiro) call MessageBoxA ; ← user32.dll / __stdcall ; A MessageBoxA termina com "RET 16" internamente — ; ela mesma retira os 16 bytes (4 args × 4B) da pilha! ; O caller NÃO precisa fazer nada depois. ; resultado (IDOK=1) em EAX xor eax, eax ret

Quem limpa a pilha no __stdcall? A callee. Internamente, MessageBoxA termina com RET 16 — isso faz o processador retirar o endereço de retorno e mais 16 bytes (os 4 args) da pilha de uma vez. O caller não precisa fazer nada depois do CALL.

04 — Comparação direta: __stdcall vs Microsoft x64

Os dois contratos lado a lado

Mesma chamada, duas convenções, resultados completamente diferentes no Assembly. Entender isso é essencial para ler código gerado pelo compilador ou depurar com o WinDbg.

Característica __stdcall (x86 32-bit) Microsoft x64 ABI (64-bit)
Passagem de args Todos na pilha, ordem inversa 4 primeiros em RCX, RDX, R8, R9; resto na pilha
Limpeza da pilha Callee — usa RET N Caller — usa ADD RSP, N
Valor de retorno EAX (32-bit) RAX (64-bit)
Shadow space ❌ Não existe ✅ Sempre 32 bytes obrigatórios
Callee-saved EBXESIEDIEBP RBXRBPRDIRSIR12–R15
Decoração de nome _MessageBoxA@16 MessageBoxA (sem decoração)
Usado em WinAPI 32-bit, COM 32-bit, DLLs legadas Toda WinAPI 64-bit, drivers, DLLs modernas
⚠️

Misturar as convenções causa stack corruption — um bug clássico difícil de depurar. Se você chamar uma __stdcall como se fosse __cdecl (onde o caller limpa), a pilha fica desalinhada e o programa trava ou produz comportamento aleatório.

05 — Exemplo avançado: CreateFileW com 7 argumentos (x64)

Quando há mais de 4 args, o resto vai na pilha

CreateFileW tem 7 argumentos. Os 4 primeiros vão em registradores, os outros 3 na pilha — acima do shadow space obrigatório.

CreateFileW — 7 argumentos em x64 (kernel32.dll)

Regra de ouro Windows x64: RCX → RDX → R8 → R9 para os 4 primeiros, depois pilha. Shadow space de 32 bytes sempre. Quem limpa é o caller com ADD RSP, N.

06 — Quem salva os registradores? (Windows x64)

Voláteis vs. Não-voláteis — o que a WinAPI pode destruir

Quando você chama MessageBoxA, ela pode sobrescrever alguns registradores livremente. Se você tiver um valor importante neles, precisa salvá-los antes do CALL.

🔴 Voláteis — Caller-saved

a WinAPI pode destruir
RAX RCX RDX R8 R9 R10 R11 XMM0–5

Se você precisar do valor de R10 depois de chamar uma WinAPI, salve com PUSH R10 antes do CALL.

🟣 Não-voláteis — Callee-saved

a WinAPI deve preservar
RBX RBP RDI RSI R12 R13 R14 R15 XMM6–15

Depois de chamar qualquer WinAPI, você garante que RBX/R12–R15 têm os mesmos valores que antes. Mas se a sua função usar eles, ela precisa salvar e restaurar.

Usando RBX corretamente entre chamadas de WinAPI
Preservando RBX entre chamadas à WinAPI — x64
; Nossa função usa RBX para guardar um handle entre chamadas processar_arquivo: push rbx ; ← SALVA RBX (obrigatório, é callee-saved) sub rsp, 40 ; shadow space + alinhamento ; Abre o arquivo e guarda o HANDLE em RBX lea rcx, nome_arquivo mov edx, 0x80000000 ; GENERIC_READ mov r8d, 1 xor r9d, r9d mov dword ptr[rsp+32], 3 mov dword ptr[rsp+40], 128 mov qword ptr[rsp+48], 0 call CreateFileW mov rbx, rax ; guarda HANDLE em RBX ; Chama MessageBoxA — pode destruir RAX, RCX, RDX, R8, R9... ; mas NÃO toca em RBX (é callee-saved na x64 ABI) xor ecx, ecx lea rdx, ok_msg lea r8, titulo xor r9d, r9d call MessageBoxA ; RBX ainda tem o HANDLE original! mov rcx, rbx ; usa o HANDLE guardado call CloseHandle ; fecha o arquivo add rsp, 40 pop rbx ; ← RESTAURA RBX antes de sair ret
💡

Use RBX, R12–R15 para guardar valores que precisam sobreviver a chamadas de WinAPI — eles são callee-saved e nenhuma função do Windows vai sobrescrevê-los. Só lembre de salvá-los no início e restaurá-los no final da sua função.
