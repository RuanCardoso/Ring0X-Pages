Entenda como funções se comunicam entre si: como passam parâmetros, retornam valores e quem é responsável por preservar os registradores.

01 — Por que precisamos de uma convenção?

Funções precisam "conversar" em uma língua comum

Quando uma função chama outra, elas precisam combinar onde os parâmetros serão colocados e onde o resultado estará. Sem um acordo, o código vira bagunça — especialmente quando C chama Assembly, ou um programa usa uma biblioteca compilada por outro compilador.

// Quem chama (caller) — main()
int resultado = soma(10, 20);

// Quem é chamado (callee) — soma()
int soma(int a, int b) {
    return a + b;  // onde colocar o retorno?
}
💡

A convenção de chamadas (calling convention) é o "contrato" que define: onde ficam os parâmetros, onde vai o valor de retorno, quem limpa a pilha, e quais registradores devem ser preservados.

02 — A Pilha (Stack): a memória das funções

A pilha cresce para baixo

A pilha é uma região especial da memória usada para guardar informações temporárias de cada função: variáveis locais, parâmetros (em alguns casos) e o endereço de retorno. Ela funciona como uma pilha de pratos — você só coloca e tira do topo.

O registrador SP (Stack Pointer) sempre aponta para o topo atual da pilha. Quando uma função começa, o SP é decrementado para reservar espaço. Quando ela termina, o SP volta ao valor anterior — "limpando" o frame.

03 — Registradores: onde os dados realmente vivem

A CPU usa registradores como "variáveis ultrarrápidas"

Antes de falar em pilha, a CPU tenta passar parâmetros e retornar valores em registradores — eles são muito mais rápidos do que qualquer acesso à memória. A convenção System V AMD64 ABI (usada no Linux/macOS 64-bit) define papéis fixos para cada registrador.

04 — O que acontece quando você chama uma função?

O ciclo completo de uma chamada de função

Veja o que ocorre nos bastidores quando soma(10, 20) é executado no nível de Assembly:

1

Caller coloca os argumentos nos registradores

MOV RDI, 10 — coloca o 1º argumento em RDI
MOV RSI, 20 — coloca o 2º argumento em RSI

2

Instrução CALL — "liga" para a função

CALL soma faz duas coisas: empurra o endereço de retorno na pilha (para saber onde voltar) e salta para o início de soma.

3

Callee cria seu Stack Frame (prólogo)

PUSH RBP — salva o frame pointer anterior
MOV RBP, RSP — define a base do novo frame
SUB RSP, N — reserva espaço para variáveis locais

4

Callee executa o corpo da função

Acessa os argumentos via RDI e RSI. Faz os cálculos. Coloca o resultado em RAX — o registrador reservado para o valor de retorno.

5

Callee desfaz o frame (epílogo)

MOV RSP, RBP — restaura o ponteiro da pilha
POP RBP — restaura o frame pointer do caller

6

Instrução RET — "desliga" e volta

RET retira o endereço de retorno da pilha e salta de volta para o caller. O caller lê o resultado em RAX.

🔁

Esse ciclo — prólogo → corpo → epílogo — se repete para cada função chamada, criando uma "pilha de frames" que reflete exatamente a cadeia de chamadas ativas no momento.

05 — Quem salva os registradores?

Caller-saved vs. Callee-saved

Quando uma função chama outra, a função chamada pode sobrescrever alguns registradores. O contrato define quem é responsável por salvar e restaurar cada registrador antes que ele seja destruído.

🔴 Caller-saved 🟣 Callee-saved
O caller (quem chama) salva antes do CALL, se precisar do valor depois. O callee (quem foi chamado) salva no início e restaura no final.
A função chamada pode destruir esses valores sem avisar. A função chamada garante que o valor estará intacto ao retornar.
Registradores: RAX RCX RDX RSI RDI R8–R11 Registradores: RBX RBP R12–R15
Analogia: você guarda sua carteira antes de deixar o estranho sentar na sua cadeira. Analogia: o visitante anota o que mexeu e recoloca tudo no lugar antes de ir.
⚠️

Se um callee usa RBX ou R12, ele deve salvá-los na pilha com PUSH RBX no início e restaurá-los com POP RBX antes do RET. Esquecer isso causa bugs difíceis de detectar.

06 — Exemplo completo em Assembly

Vendo C e Assembly lado a lado

O compilador transforma a chamada int r = soma(10, 20) para algo assim em x86-64:

CÓDIGO C
int soma(int a, int b) {
    return a + b;
}

int main() {
    int r = soma(10, 20);
    return r;
}
ASSEMBLY GERADO (x86-64)
soma:
  ; prólogo
  push  rbp
  mov   rbp, rsp
  ; corpo: a está em EDI, b em ESI
  lea   eax, [rdi + rsi]
  ; epílogo
  pop   rbp
  ret            ; resultado em EAX

main:
  push  rbp
  mov   rbp, rsp
  mov   edi, 10   ; 1º arg → RDI
  mov   esi, 20   ; 2º arg → RSI
  call  soma
  ; resultado agora em EAX
  pop   rbp
  ret
